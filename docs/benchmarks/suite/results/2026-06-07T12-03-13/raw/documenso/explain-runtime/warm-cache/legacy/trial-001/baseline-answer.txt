## How Documenso moves a document through its lifecycle

Documenso models documents as **`Envelope`** rows (`EnvelopeType.DOCUMENT`), with child `Recipient` and `Field` rows and a `DocumentMeta`. Status flows `DRAFT → PENDING → COMPLETED` (or `REJECTED`). The pipeline spans four stages, each in its own server module, with email side-effects pushed onto the background **job queue** (`packages/lib/jobs/client.ts`).

### 1. Recipient creation (send preparation)
`packages/lib/server-only/recipient/create-envelope-recipients.ts`
- Validates the envelope isn't already complete and that action-auth is permitted, then in a transaction creates each `Recipient` with a unique `token` (`nanoid()`), `sendStatus = NOT_SENT`, `signingStatus = NOT_SIGNED`.
- **CC recipients are special-cased**: created already `SENT`/`SIGNED` since they take no action.
- Writes a `RECIPIENT_CREATED` audit log (`createDocumentAuditLogData`).

### 2. Sending / status transition
`packages/lib/server-only/document/send-document.ts`
- Loads the envelope with recipients/fields/meta/items, enforces guards: non-disabled user, has recipients, recipient-count limit, not already completed.
- Validates recipients with auth requirements have valid emails (`isRecipientEmailValidForSending`) and that signers have required fields (`getRecipientsWithMissingFields`); for V2 envelopes computes auto-insertable field values (`extractFieldAutoInsertValues`).
- Computes `recipientsToNotify`: **all** recipients for `PARALLEL`, or just the first unsigned non-CC for `SEQUENTIAL` signing order (`DocumentSigningOrder` / `documentMeta.signingOrder`).
- In a `prisma.$transaction`: writes `DOCUMENT_SENT` audit log, auto-inserts V2 fields, stamps recipient `expiresAt`, and flips `Envelope.status` **`DRAFT → PENDING`**.
- If everyone is CC/already-signed, it short-circuits straight to the `internal.seal-document` job.

### 3. Notification delivery
Triggered from `sendDocument` as the `send.signing.requested.email` job (skipping CC and already-`SENT` recipients), handled by:
`packages/lib/jobs/definitions/emails/send-signing-email.handler.ts`
- Re-checks the envelope is `PENDING`, email settings enabled (`extractDerivedDocumentEmailSettings`), org email rate limits (`assertOrganisationRatesAndLimits`).
- Renders `DocumentInviteEmailTemplate` with the signing link `/sign/{recipient.token}`, sends via `emailTransport` (`getEmailContext`), then updates the recipient to `sendStatus = SENT` + `sentAt`, schedules the first reminder (`updateRecipientNextReminder`), and logs `EMAIL_SENT`.
- `sendDocument` separately fires the `DOCUMENT_SENT` webhook (`triggerWebhook`).

### 4. Signing state + downstream notifications
`packages/lib/server-only/document/complete-document-with-token.ts` (recipient hits `/sign/{token}`)
- Loads the recipient by token, asserts envelope `PENDING`, not expired (`assertRecipientNotExpired`), not already signed/rejected, and—if `SEQUENTIAL`—that it's their turn (`get-is-recipient-turn.ts`).
- Enforces access 2FA (`is-recipient-authorized.ts`), auto-inserts date fields (V2), and rejects if required fields remain unsigned (`fieldsContainUnsignedRequiredField`).
- Transaction: sets recipient `signingStatus = SIGNED` + `signedAt`, with audit logs (`DOCUMENT_RECIPIENT_COMPLETED`).
- Side effects: `DOCUMENT_RECIPIENT_COMPLETED` webhook + `send.recipient.signed.email` job (owner notification).
- If recipients still pending → `send-pending-email.ts` ("waiting for others"); for `SEQUENTIAL`, marks the **next** recipient `SENT` and fires another `send.signing.requested.email`.
- When **all** non-CC recipients are signed → triggers `internal.seal-document`, then fires the `DOCUMENT_SIGNED` webhook.

### 5. Completion / sealing
`packages/lib/jobs/definitions/internal/seal-document.handler.ts`
- Marks CC recipients signed, verifies completeness, flattens + inserts fields into the PDF (`insertFieldInPDFV1/V2`), appends certificate + audit-log PDFs (`generateCertificatePdf` / `generateAuditLogPdf`), and digitally signs it (`@documenso/signing` `signPdf`).
- Sets `Envelope.status = COMPLETED` (or `REJECTED`) + `completedAt`, writes `DOCUMENT_COMPLETED`, fires the `DOCUMENT_COMPLETED`/`DOCUMENT_REJECTED` webhook, and enqueues `send.document.completed.emails` to deliver the final signed copy.

### Key files
- Recipients: `create-envelope-recipients.ts`
- Send/prepare + status flip: `document/send-document.ts`
- Invite email job: `jobs/definitions/emails/send-signing-email.handler.ts`
- Signing: `document/complete-document-with-token.ts` (+ `recipient/get-is-recipient-turn.ts`, `document/send-pending-email.ts`)
- Sealing/finalize: `jobs/definitions/internal/seal-document.handler.ts`
- Webhooks: `server-only/webhooks/trigger/trigger-webhook.ts`; job dispatch: `jobs/client.ts`

The connective tissue is: **DB status transitions happen synchronously in `send-document`/`complete-document-with-token`, while all email/notification delivery is dispatched asynchronously through the job queue, and every state change emits both an audit log and a webhook.**
