## Overview

A survey response flows through three stages: **(1) synchronous request handling + validation**, **(2) transactional persistence with quota evaluation**, and **(3) asynchronous side-effects** (webhooks, integrations, emails, analytics/event tracking) dispatched via a background job pipeline.

## 1. Request handling & validation

**File: `apps/web/app/api/v2/client/[workspaceId]/responses/route.ts`** — the `POST` handler hit by the survey widget/SDK:

1. `resolveClientApiIds` resolves the path param (accepts either a legacy `environmentId` or new `workspaceId`).
2. `parseAndValidateResponseInput` parses the JSON body and validates against `ZResponseInputV2`.
3. `getContactsDisabledResponse` enforces the enterprise license check for `contactId` (user identification).
4. `getSurvey(surveyId)` (`@/lib/survey/service`) loads the survey; 404 if missing.
5. `validateResponseSubmission` runs the gauntlet:
   - `checkSurveyValidity` (`./lib/utils`)
   - `validateClientFileUploads` (file-upload questions)
   - `validateOtherOptionLengthForMultipleChoice` (character limits on "other")
   - `validateResponseData` (`@/modules/api/lib/validation`) against survey blocks/questions/language
6. `createResponseForRequest` enriches metadata: parses the User-Agent via `UAParser` (browser/device/OS), derives `country` from `CF-IPCountry`/`CloudFront-Viewer-Country` headers, and—only if `survey.isCaptureIpEnabled`—captures the IP via `getClientIpFromHeaders`.

## 2. Persistence (transactional + quota evaluation)

**File: `apps/web/app/api/v2/client/[workspaceId]/responses/lib/response.ts`** → wraps **`apps/web/app/api/client/[workspaceId]/responses/lib/response.ts`**:

- `createResponseWithQuotaEvaluation` opens a single **`prisma.$transaction`**:
  1. `createResponse` (the v2 lib) validates inputs, resolves org/contact, computes time-to-completion (`calculateTtcTotal`), and asserts display ownership (`assertDisplayOwnership`) if a `displayId` is present.
  2. `buildPrismaResponseData` constructs the Prisma `ResponseCreateInput` (connects survey/display/contact, stores `data`, `meta`, `variables`, `ttc`, `singleUseId`), then `prisma.response.create` persists it with `responseSelection`.
  3. Prisma error mapping converts unique-constraint violations into `InvalidInputError`/`UniqueConstraintError` (e.g. display already linked, single-use link reused).
  4. **In the same transaction**, `evaluateResponseQuotas` (`@/modules/ee/quotas/lib/evaluation-service`) runs, returning a possible `quotaFull`.
- The route strips `quotaFull` and returns just `{ id, ...quotaObj }` to the client.

## 3. Pipeline dispatch → async side-effects / analytics

Back in the route, after persistence:

- `sendToPipeline({ event: "responseCreated", ... })` is always called; if `responseData.finished`, a second `responseFinished` event is sent.
- **File: `apps/web/app/lib/pipelines.ts`** — `sendToPipeline` enqueues onto **BullMQ** via `getBackgroundJobProducer().enqueueResponsePipeline` (from `@formbricks/jobs`).

The job is consumed by the registered app override (the stub `packages/jobs/src/processors/response-pipeline.ts` is replaced by):

**File: `apps/web/modules/response-pipeline/lib/process-response-pipeline-job.ts`** — `processResponsePipelineJob`:

1. Loads organization, survey, and matching webhooks in parallel; validates survey↔workspace ownership (throws `UnrecoverableError` otherwise).
2. **Webhooks** (`deliverWebhooks` / `createWebhookDeliveryTask`): builds a signed payload (Standard Webhooks signature), resolves storage URLs, and POSTs with SSRF protections (URL validation, pinned-IP dispatcher, manual-redirect blocking, 5s timeout). Failures retry until `maxAttempts`.
3. On **`responseFinished`** (`runResponseFinishedSideEffects`):
   - `handleIntegrations` (Google Sheets, Airtable, Notion, Slack, etc.)
   - `handleFeedbackSourcePipeline`
   - `sendFollowUpsForResponse` (follow-up emails)
   - `sendResponseFinishedEmail` to org members/teams with alert notifications enabled
   - `handleSurveyAutoCompleteSafely` — marks survey `completed` when `responseCount` ≥ `autoComplete`, with an audit-log event.
4. On **`responseCreated`** (`runResponseCreatedSideEffects`) — the analytics/event-tracking stage:
   - `recordResponseCreatedMeterEvent` (Stripe billing metering)
   - **`captureSurveyResponsePostHogEvent`** (`./lib/posthog.ts`) → `capturePostHogEvent` emits the `survey_response_received` PostHog event, but only at milestones (1st, every 10th up to 100, then every 100th)
   - `sendTelemetryEvents` (self-hosted telemetry)

## Key files

- `apps/web/app/api/v2/client/[workspaceId]/responses/route.ts` — request handling, validation, metadata, pipeline dispatch
- `apps/web/app/api/v2/client/[workspaceId]/responses/lib/response.ts` & `apps/web/app/api/client/[workspaceId]/responses/lib/response.ts` — persistence + quota in one transaction
- `apps/web/modules/ee/quotas/lib/evaluation-service.ts` — quota evaluation
- `apps/web/app/lib/pipelines.ts` + `@formbricks/jobs` — BullMQ enqueue
- `apps/web/modules/response-pipeline/lib/process-response-pipeline-job.ts` — async webhooks/integrations/emails/auto-complete
- `apps/web/modules/response-pipeline/lib/posthog.ts` — analytics/event tracking (`survey_response_received`)

One notable design point: analytics and all third-party effects are intentionally **out of the request path** — the client gets a fast response after the DB transaction, and everything else runs in a retryable background job where individual failures are logged and isolated rather than failing the submission.
