# 2026-06-01 design-partner workflow loops

This folder publishes three **anonymized** workflow-loop drafts that stand in for the kind of customer-style proof Madar still needs.

They are intentionally **synthetic reproductions** based on repo-owned workflows and public tooling surfaces, not claims that five live design partners already ran these exact loops. Sensitive source, prompts, and customer details are excluded. The goal is to show what a repeatable publishable workflow note should look like before the repo has more live partner receipts.

## Included drafts

- [`01-explain-flow.md`](01-explain-flow.md) — explain-flow loop for an anonymized repo owner trying to answer a bounded runtime question with a share-safe summary.
- [`02-review-pr.md`](02-review-pr.md) — review-PR loop showing how Madar-backed diff evidence can feed a reviewer and preserve human follow-up notes.
- [`03-impact-change.md`](03-impact-change.md) — impact-change loop for blast-radius planning before a refactor or risky code move.

## Safety boundary

- Keep the published notes **anonymized**.
- Use **synthetic reproductions** or partner-approved summaries when raw repos cannot be shared.
- Exclude sensitive source, full prompts, and customer details from the public note.
- Prefer share-safe artifacts such as `report.share-safe.json`, short workflow summaries, and bounded command receipts.

## Safe interpretation

This bundle is useful because it makes the workflow-proof target concrete, but it is **not yet five live design partners**. It does not prove universal win rates, and it does not justify broad public claims about customer adoption. It is a draftable publication format for future workflow evidence.
